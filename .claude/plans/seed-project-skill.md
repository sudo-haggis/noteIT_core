# Plan: `/seed-project` skill — idea-to-PARA-project conversion

## Context

After brainstorming conversations with Claude, the user captures ideas in a freeform document. Currently there is no workflow to convert that document into a proper PARA project structure. This plan introduces:

- `_briefs/` — a capture folder at vault root for post-conversation idea docs
- `/seed-project` — a skill that reads a brief and scaffolds a hierarchical PARA project with sequential phase notes
- `sub-projects:` — a new optional frontmatter field on parent project index notes
- An addition to `note-schema.md` documenting the sub-folder pattern

The user confirmed they want **freeform parsing** — Claude reads the brief and interprets it intelligently rather than requiring strict structure.

---

## New Concept: Idea Brief files

Briefs live in `_briefs/` at vault root (alongside `INBOX.md`, outside PARA).

- Evergreen brief: `_briefs/<slug>.md`
- Timestamped: `_briefs/YYYY-MM-DD_<slug>.md`
- Template: `_briefs/_template.md` (canonical reference — skill must never process this)

**Recommended light structure (not enforced):**
```markdown
---
workspace: Acme
priority: high
---

# Project Title

## Goal
One paragraph — what does success look like?

## Phases
- Phase one name
- Phase two name
- Phase three name

## Context
Freeform notes from the conversation.
```

The skill uses Claude's intelligence to extract workspace, priority, title, phases, goal, and context from whatever the user wrote — numbered lists, prose, bullet points all work.

## Output structure generated by the skill

```
<workspace>/projects/<project-slug>/
  index.md                   ← parent project (overview + wikilinks)
  01-<phase-1-slug>.md       ← Phase 1
  02-<phase-2-slug>.md       ← Phase 2
  03-<phase-3-slug>.md       ← Phase 3
```

- Parent is always `index.md` (never named after the slug)
- Phase files zero-padded two digits: `01-`, `02-`, ... `09-`, `10-`
- Phase slugs: kebab-case from phase name, strip any leading "Phase N:" prefix the user typed

## Skill behaviour: `/seed-project <path-to-brief>`

### Steps the skill executes

**1. Read and parse brief**
- Extract from frontmatter: `workspace`, `priority` (default `medium` if absent)
- Extract H1 as project title; if no H1, derive from filename
- Extract phases from any list-style section (numbered, bulleted, or prose enumeration)
- Extract goal and context text if present
- **Abort** if: `workspace` missing, workspace folder doesn't exist, no phases found, path is `_template.md`

**2. Derive slugs**
- Project slug: lowercase H1, hyphens only, no special chars
- Phase slugs: same rule, strip leading "Phase N:" / "Step N" prefixes

**3. Collision check**
- If `<workspace>/projects/<project-slug>/` already exists → abort without touching files
- If flat `<workspace>/projects/<project-slug>.md` exists → warn and prompt to continue

**4. Preview and confirm**
Show the full file list + dependency chain before writing anything. Wait for `y/n`.

**5. Write `index.md`**
```yaml
sub-projects:
  - <workspace>/projects/<project-slug>/01-<phase-1-slug>
  - ...
```
Body includes `## Goal`, `## Phases` (wikilinks with labels `Phase N: <name>`), `## Context` (if present), and the standard `> [!info]- Dependencies` callout.

**6. Write phase notes** (one per phase, in order)

Phase 1 `depends-on`: `[index]`
Phase N `depends-on`: `[index, 0(N-1)-<prev-slug>]`

Body callout mirrors `depends-on` exactly — one wikilink per path (dual-representation rule).

**7. Confirm output** — list files created, dependency chain summary, and next-steps hints.
