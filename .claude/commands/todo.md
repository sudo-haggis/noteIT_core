# Command: todo

Scan the vault for inline TODO markers, render them grouped by project (with a
priority-sorted "Urgent" glance at the top), and write the output to a Markdown
file for Obsidian viewing and git sync.

## Usage

```
/todo [--workspace <name>] [--priority <N>] [--top]
```

Examples:
```
/todo                      # Scan entire vault → writes TODO.md at vault root
/todo --workspace Acme      # Scan Acme only    → writes Acme/TODO.md
/todo --priority 1         # Show only P1 items (still writes output file)
/todo --top                # Show only the highest populated tier
```

---

## Marker format (for reference)

Canonical form is checkbox-prefixed — see `.claude/skills/note-schema.md`'s
"Inline TODO Markers" section for the full convention:

| Marker | Priority | Output tag |
|--------|----------|------------|
| `- [ ] TODO-1: message` | Highest | `[P1]` |
| `- [ ] TODO-2: message` | Second | `[P2]` |
| `- [ ] TODO-N: message` | Nth | `[PN]` |
| `- [ ] TODO: message` | Unranked | `[--]` |

Ticking the checkbox (`- [x]`) retires the marker — `toDoIt` excludes it before
`/todo` ever sees it, so nothing extra is needed in the steps below. Bare markers
(no checkbox, e.g. `TODO-1: message`) still work identically to before, just
without the tick-to-resolve option.

---

## What to do

### Step 1 — Determine scan root and output file path

**No `--workspace` flag:**
- Scan root = vault root
- Output file = `TODO.md` at vault root

**`--workspace <name>` given:**
- Check the folder exists → if not, abort: "Workspace '<name>' not found."
- Scan root = `<name>/` relative to vault root
- Output file = `<name>/TODO.md`

### Step 2 — Run toDoIt

For vault-wide scan, run from vault root:
```bash
cd <vault-root> && toDoIt -d
```

For workspace-scoped scan, run from the workspace subdirectory:
```bash
cd <vault-root>/<workspace> && toDoIt -d
```

The vault root `.todoignore` excludes `.obsidian/`, `.claude/`, `.git/`, `_briefs/` for
vault-wide scans. Workspace-scoped scans don't use `.todoignore` — that's fine, those
directories are not present inside workspace folders.

### Step 3 — Parse the output

Each hit line from toDoIt has this format (with ANSI colour codes — strip them):
```
[PN] line <N> — <full line content including TODO-N: prefix>
```

The file path appears in the separator block above the hits:
```
========================================
./path/to/file.md
========================================
```

For each hit, record:
- `file`: path from nearest header above (strip leading `./`)
- `line`: the line number
- `priority`: the number from `[PN]`, or `99` for `[--]` (unranked, sorts last)
- `message`: text after the `TODO-N:` or `TODO:` prefix on the hit line
- `note_title`: derive from the filename — strip path prefix, strip `.md`, convert hyphens and
  underscores to title-cased words. E.g. `01-pi-foundation.md` → `Pi Foundation`
- `workspace`: first path segment of `file` (e.g. `linux_system` from
  `linux_system/projects/.../01-pi-foundation.md`)

**Discard any hits from `TODO.md` files** — these are previously generated output files,
not source notes.

### Step 4 — Apply filters

- `--priority <N>` → keep only hits where `priority == N`
- `--top` → find the lowest priority number present, keep only those hits
- No filter → keep all hits

### Step 5 — Render output

Build the output as a Markdown string, grouped by **project first**, priority within
each project — not priority first. A priority-first layout interleaves unrelated
projects under each tier, which makes it hard to focus on one project at a time.

An **Urgent** section at the top gives a flat, priority-sorted glance across
everything (P1-P3 only) before the full per-project breakdown. Only include the
Urgent section on an **unfiltered** run (no `--priority`, no `--top`) — if a filter
is already active, the whole output is already a narrow slice, so a separate Urgent
section would just duplicate it.

**Unfiltered vault-wide output (`TODO.md` at vault root):**
```markdown
---
title: "TODO List"
updated: <today YYYY-MM-DD>
---

# TODO List
> Generated by `/todo` — source of truth is in the notes. Do not edit manually.

## Urgent (P1-P3)
- [P<N>] [[<file-path-no-ext>|<workspace> / <note_title>]] (line <N>) — <message>

## By Project

### <workspace> / <note_title>
- [P<N>] (line <N>) — <message>
- [--] (line <N>) — <message>
```

**Unfiltered workspace output (`<workspace>/TODO.md`):**
```markdown
---
title: "<workspace> — TODO List"
updated: <today YYYY-MM-DD>
---

# <workspace> — TODO List
> Generated by `/todo --workspace <workspace>` — source of truth is in the notes. Do not edit manually.

## Urgent (P1-P3)
- [P<N>] [[<file-path-no-ext>|<note_title>]] (line <N>) — <message>

## By Project

### <note_title>
- [P<N>] (line <N>) — <message>
- [--] (line <N>) — <message>
```

**Filtered output** (`--priority <N>` or `--top` given, either scope) — omit the
Urgent section entirely, keep just `## By Project` with the same per-project
grouping. Workspace subheading in project headings is omitted for workspace-scoped
runs (workspace is implicit), same as before.

**Rules for rendering:**
- **Urgent section**: flat list, no subheadings, sorted by priority ascending
  (P1, P2, P3), then by the order toDoIt found the items within each priority.
  Vault-wide runs label each link `<workspace> / <note_title>` — several notes
  across the vault share the same title (e.g. multiple `readme-audit.md`s), and
  with no subheadings to disambiguate, workspace has to be in the label itself.
  Workspace-scoped runs just use `<note_title>` (no collision risk within one
  workspace).
- **By Project section**: one `###` heading per source note that has at least one
  hit. Vault-wide heading format is `<workspace> / <note_title>`; workspace-scoped
  format is just `<note_title>`. Sort headings alphabetically by workspace, then
  alphabetically by note_title within it — this section is for browsing, not
  urgency, so keep the ordering predictable rather than re-sorting by priority.
- Within a project's own list: sort by priority ascending, unranked (`[--]`) last,
  preserving toDoIt's found order for ties.
- Wikilink path = vault-relative file path without `.md` extension
- Wikilink label = `note_title` derived from filename
- If no hits at all after filtering: write only the header and a single line
  `> Nothing to do! ✓`

### Step 6 — Write the output file

Write the rendered Markdown to the output file path determined in Step 1.
Overwrite the file completely on every run — it is a generated view, not a document.

### Step 7 — Print to terminal

After writing the file, print the same rendered output to the terminal so the user
can see the result without opening the file.

Then print:
```
Written to: <output-file-path>
```

---

## Rules

- Always run `toDoIt` with the `-d` flag (no depth limit)
- Never modify vault notes — this is a read-only scan (only the output TODO.md is written)
- Output files are always fully overwritten — never appended to
- Discard hits from any file named `TODO.md` to avoid scanning previous output
- Wikilink paths are vault-relative with no `.md` extension and no leading `./`
- Unranked items always appear last
- The output file has minimal frontmatter (`title` and `updated` only — no `para`, `status`,
  etc.) to distinguish it as a generated view, not a PARA note
